## 🟠 High Priority (This Week)

### 1. Bench Player Names Not Escaped in Detail String
**File:** `Code.gs`
**Lines:** 1533–1539
**Risk:** Medium (HTML injection in personal email)

When building the bench summary callout, player names from `benchSummary.topBench` are joined into the `detail` string without calling `escapeHtml_()`. That `detail` string is then directly concatenated into the HTML output at line 1539.

```javascript
// Line 1533-1535 — bp.name is unescaped
detail = benchSummary.topBench.map(function (bp) {
  return bp.name + ' ' + bp.points.toFixed(1);  // <-- bp.name not escaped
}).join(', ');
```

The resulting string is injected into the email body here:
```javascript
// Line 1539
(detail ? ' (' + detail + ')' : '')
```

A player with `<` or `>` in their name (uncommon but possible via API data) would corrupt the HTML structure. A player named "T.J. Hockenson & Co." would produce a literal `&` in the rendered output (though `&` alone doesn't break rendering, it is technically invalid HTML).

**Recommended Fix:** Escape `bp.name` when building the detail string:
```javascript
detail = benchSummary.topBench.map(function (bp) {
  return escapeHtml_(bp.name) + ' ' + bp.points.toFixed(1);
}).join(', ');
```

**Why High Priority:** This is the single remaining unescaped HTML injection point after the March 2026 fix pass. It is a small fix that completes the `escapeHtml_()` coverage to 100%.

## 🟡 Medium Priority (Next Sprint)

### 2. `verifySupabaseSchema_()` Does Not Block Execution on Failure
**File:** `Code.gs`
**Lines:** 1797–1799
**Risk:** Silent data-loss when schema is missing

`verifySupabaseSchema_()` is called at the start of `pullFantasyData()` when Supabase is configured, which is correct. However, the function returns `false` on failure and only logs a message — it does not prevent the run from proceeding. `persistWeeklySnapshot_()` will then be called inside `buildLeagueSnapshot_()` for each league, fail silently (the error is caught and logged), and leave a permanent data gap in the Supabase table that cannot be backfilled without a manual re-run.

```javascript
// pullFantasyData, lines 1797-1799
if (isSupabaseConfigured_()) {
  verifySupabaseSchema_();  // return value not checked; failure is ignored
}
```

There are two reasonable behaviors: (a) abort early and send an error email, or (b) continue the run but send a warning notification email so the user knows Supabase persistence is broken. Option (b) is less disruptive. Currently, neither happens.

**Recommended Fix:** Check the return value and send a notification email if the schema is missing:
```javascript
if (isSupabaseConfigured_()) {
  if (!verifySupabaseSchema_()) {
    sendNotificationEmail_('Fantasy Snapshot - Supabase Schema Missing',
      'weekly_snapshots table is not accessible. Snapshot will run but season data will NOT be persisted. ' +
      'Run the SQL schema from CLAUDE.md and verify RLS policies are set.');
  }
}
```

**Why Medium:** The email still sends and the snapshot is still useful. But silent Supabase failures create confusing data gaps that are hard to debug later.

### 5. `getWeeklyPowerRankings_()` Makes Up to 3 Serial API Calls With No Short-Circuit
**File:** `Code.gs`
**Lines:** 738–742
**Risk:** Execution time / quota

`getWeeklyPowerRankings_()` calls `getWeekTeamScores_()` for each week in the rolling window (up to 3 calls), then separately calls `getWeekTeamHighlights_()` just a few lines later in `buildLeagueSnapshot_()`. Both functions hit `/league/{key}/scoreboard;week=N`. For `completedWeek`, both calls fetch the same scoreboard payload.

Specifically:
- `getWeeklyPowerRankings_(completedWeek)` → calls `getWeekTeamScores_(completedWeek)` (line 739)
- `getWeekTeamHighlights_(completedWeek)` → calls `/scoreboard;week=completedWeek` (line 595)

These are separate HTTP requests for the identical endpoint and week. For a single-league account this is 1 extra API call per run. For a 3-league account it is 3 extra calls.

**Recommended Fix (low risk):** Pass the already-fetched `completedWeek` scores from `getWeeklyPowerRankings_` into `getWeekTeamHighlights_`, or extract the scoreboard fetch into a shared helper. Alternatively, flag this as a known tradeoff in a comment.

**Note:** This issue existed before the March 2026 fixes but was not previously tracked. It is a smaller impact than the duplicate roster fetch (which was fixed) and is explicitly carried forward here.
