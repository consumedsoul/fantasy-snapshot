## 🟠 High Priority (This Week)

### 1. Season Trends Always Lag by One Completed Week
**File:** `Code.gs`
**Lines:** 1398–1460
**Risk:** Data freshness — user sees stale trend data

In [buildLeagueSnapshot_](Code.gs:1358), `getSeasonTrends_()` is invoked at line 1400, which queries Supabase. The current-completed-week persist call (`persistWeeklySnapshot_`) happens later at line 1456. On the first run after a week completes, Supabase has not yet received the current completed week's row, so the Season Trends section builds its rolling averages, consistency, and luck calculations from data that is one week behind the rest of the email.

Concretely: on the Monday after Week 10 concludes, the email's "Week 10 Highlights" section shows Week 10 correctly, but the "Season Trends" table reflects data through Week 9 only. The user has no signal this is happening.

**Current order:**
```javascript
// Line 1400 — reads Supabase (misses current completed week)
var seasonTrends = getSeasonTrends_(leagueKey);
// ...
// Line 1456 — writes current completed week
persistWeeklySnapshot_(leagueKey, completedWeek, rows, weeklyScoreMap);
```

**Recommended Fix:** Reorder so the persist happens before the trends query. Move the completed-week scoreboard fetch + persist block above the `getSeasonTrends_` call:

```javascript
// Fetch completed-week data first
var completedWeek = currentWeek - 1;
var sharedScoreboard = fetchWeekScoreboard_(completedWeek, leagueKey);
var sharedWeekScores = getWeekTeamScores_(completedWeek, leagueKey, sharedScoreboard);

// Persist BEFORE querying trends
if (isSupabaseConfigured_() && sharedWeekScores) {
  try {
    var weeklyScoreMap = {};
    sharedWeekScores.forEach(function (t) { weeklyScoreMap[t.team_name] = t.points; });
    persistWeeklySnapshot_(leagueKey, completedWeek, rows, weeklyScoreMap);
  } catch (persistErr) {
    Logger.log('[buildLeagueSnapshot_] Supabase persistence failed: ' + persistErr.message);
  }
}

// Now trends include current completed week
var seasonTrends = getSeasonTrends_(leagueKey);
```

Note this reorders relative to the standings table (which currently renders between standings and trends). The standings table rendering does not depend on scoreboard/persist, so nothing breaks.

**Why High Priority:** Real data-correctness issue visible in every weekly email — unlike a one-off escaping miss, this affects every run once the season has ≥3 weeks of data.

### 4. `escapeHtml_()` Does Not Escape Single Quotes
**File:** `Code.gs`
**Lines:** 1625–1631
**Risk:** Low today, latent — breaks if any future code inserts a value into a single-quoted attribute

```javascript
function escapeHtml_(str) {
  return String(str == null ? '' : str)
    .replace(/&/g, '&amp;')
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;')
    .replace(/"/g, '&quot;');
  // No replacement for '
}
```

Every current insertion is either element content (fine) or a double-quoted attribute (fine). But if someone later writes `<td style='color:' + escapeHtml_(x) + '\'...'>`, a single quote in `x` would break out of the attribute.

**Recommended Fix:** Add a single-quote replacement for defense-in-depth:
```javascript
.replace(/'/g, '&#39;');
```

**Why Medium:** Not a live bug, but `escapeHtml_` is the one trusted choke point for all HTML output. It should be fully defensive even if current call sites are safe, because the cost of a future mistake elsewhere is another round of injection hunting.

### 5. `runTests()` Needs a Float-Tolerance Helper (Carried Over)
**File:** `Code.gs`
**Lines:** 2196–2276
**Risk:** Low — limits test expansion

`assert()` uses `===`, which is fine for strings and integers but will fail on any float comparison (e.g., `avg_weekly_score`, `std_dev`). When more tests are added for `getSeasonTrends_` or power-rankings math, they will need a tolerance helper.

**Recommended Fix:**
```javascript
function assertApprox(description, actual, expected, tolerance) {
  tolerance = tolerance || 0.01;
  if (Math.abs(actual - expected) <= tolerance) {
    Logger.log('PASS: ' + description);
    passed++;
  } else {
    Logger.log('FAIL: ' + description + ' | expected: ' + expected + ' ± ' + tolerance + ' | got: ' + actual);
    failed++;
  }
}
```
