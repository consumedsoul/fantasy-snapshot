## 🟡 Medium Priority (Next Sprint)

### Plain Text Email Fallback Is Simplistic
**File:** `Code.gs`
**Lines:** 1286
**Risk:** The plain text fallback strips HTML with a simple regex (`/<[^>]*>/g`), but doesn't insert newlines between table rows or block elements. HTML tables collapse into a single run-on line of text in plain text mode.

**Current Code:**
```javascript
var plainText = htmlBody.replace(/<[^>]*>/g, '').replace(/&mdash;/g, '—').replace(/&#9733;/g, '*').replace(/&amp;/g, '&');
```

**Recommended Fix:**
```javascript
var plainText = htmlBody
  .replace(/<\/tr>/g, '\n')
  .replace(/<\/p>/g, '\n')
  .replace(/<\/h[1-6]>/g, '\n\n')
  .replace(/<br\s*\/?>/g, '\n')
  .replace(/<hr[^>]*>/g, '\n---\n')
  .replace(/<[^>]*>/g, '')
  .replace(/&mdash;/g, '—')
  .replace(/&#9733;/g, '*')
  .replace(/&amp;/g, '&')
  .replace(/\n{3,}/g, '\n\n')
  .trim();
```

**Why:** Email clients that don't support HTML (rare but possible) will show unreadable text. The fallback should produce reasonable line breaks.

### `flattenYahooMeta_` Silently Ignores Multi-Key Entries
**File:** `Code.gs`
**Lines:** 1040-1044
**Risk:** The function only flattens entries where `Object.keys(entry).length === 1`. If Yahoo API changes to include objects with multiple keys, data would be silently dropped.

**Current Code:**
```javascript
var keys = Object.keys(entry || {});
if (keys.length === 1) {
  meta[keys[0]] = entry[keys[0]];
}
```

**Recommended Fix:** Add a warning log for unexpected entries:
```javascript
if (keys.length === 1) {
  meta[keys[0]] = entry[keys[0]];
} else if (keys.length > 1) {
  Logger.log('[flattenYahooMeta_] Unexpected multi-key entry: ' + JSON.stringify(entry));
  keys.forEach(function(k) { meta[k] = entry[k]; });
}
```

**Why:** Silent data loss is the hardest bug to diagnose. A log message makes it visible.
