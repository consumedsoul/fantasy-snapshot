## 🟠 High Priority (This Week) — ✅ RESOLVED THIS COMMIT

### 1. Position Leaders Re-Fetched All Player Stats Once Per Position (6× redundant)
**File:** `Code.gs`
**Lines (pre-fix):** 1073–1149 (`getTopPlayersForWeekAndPosition_`), called in a loop at 1576–1577
**Risk:** Cost / performance — largest single API-cost hotspot; multi-league runs risk the 6-minute GAS execution ceiling

`buildLeagueSnapshot_` iterated `['QB','WR','RB','TE','K','DEF']` and called `getTopPlayersForWeekAndPosition_` once **per position**. Each call paged through *every* rostered player key (`Object.keys(ownerMap)`, ~150–190 players for a 12-team league) in 25-key batches, fetching identical `/players;player_keys=…/stats;type=week` payloads, only to discard everyone whose `display_position` didn't match. Result: ~8 batches × 6 positions ≈ **48 Yahoo API calls** where **8** suffice — the same payloads fetched six times.

**Current Code (pre-fix):**
```javascript
var positions = ['QB', 'WR', 'RB', 'TE', 'K', 'DEF'];
positions.forEach(function (pos) {
  var topPlayers = getTopPlayersForWeekAndPosition_(completedWeek, pos, 3, ownerMap, leagueKey);
  // … fetches ALL player stats again for each pos …
});
```

**Applied Fix:** Replaced with `getTopPlayersByPositionForWeek_(week, positions, limit, ownerMap, leagueKey)` — a single batched pass that buckets each player into a `position → []` map, then sorts/caps per requested position. The caller now fetches once:
```javascript
var positions = ['QB', 'WR', 'RB', 'TE', 'K', 'DEF'];
var topByPosition = getTopPlayersByPositionForWeek_(completedWeek, positions, 3, ownerMap, leagueKey);
positions.forEach(function (pos) {
  var topPlayers = topByPosition[pos];
  // … render …
});
```
Filtering semantics preserved exactly (`!wanted[displayPosition] || !name || points === 0` mirrors the old `displayPosition !== position || !name || points === 0`). Rate-limit `Utilities.sleep` between batches retained. **~40 fewer Yahoo API calls per league per run.**

## 🟡 Medium Priority (Next Sprint)

### 2. `currentWeek` NaN Slips Past the Season-Not-Started Guard — ✅ FIXED THIS COMMIT
**File:** `Code.gs`
**Lines:** 490 (`getCurrentWeek_`), 1460 (guard in `buildLeagueSnapshot_`)
**Risk:** Degraded output — malformed email instead of a clean "season not started" message

`getCurrentWeek_` returns `parseInt(meta.current_week || meta.matchup_week, 10)`, which is `NaN` if Yahoo omits both fields. The guard `if (currentWeek === null || currentWeek < 2)` does not catch `NaN` (`NaN < 2` is `false`, `NaN === null` is `false`), so execution fell through to `completedWeek = currentWeek - 1` (= `NaN`), producing a "Week null Highlights" header and a downstream `validateWeek_` error block.

**Applied Fix:** `if (currentWeek === null || isNaN(currentWeek) || currentWeek < 2)` — a NaN week now takes the clean season-not-started path.

### 3. Transient Yahoo/Supabase 429 & 5xx Were Never Retried — ✅ FIXED THIS COMMIT
**File:** `Code.gs`
**Lines:** 58–60 (`supabaseRequest_`), 2116–2118 (`yahooApiRequest_`)
**Risk:** Resilience — a single rate-limit or upstream blip aborts the whole snapshot with no backoff

Both wrappers call `UrlFetchApp.fetch` with `muteHttpExceptions:true`, so HTTP errors return a response object **without throwing**. `retryWithBackoff_` only catches *thrown* errors, so it retried transport failures but **never** 429/503/500 — the exact transient cases backoff exists for.

**Applied Fix:** The `retryWithBackoff_` closures now inspect `getResponseCode()` and `throw` on `429` or `>= 500`, engaging the existing exponential backoff (2s/4s/8s). The Yahoo `401 token_expired` refresh path is untouched (401 ∉ {429, ≥500}, so it still falls through to the token-refresh branch as before).

### 7. No `LockService` Around Token Refresh / `pullFantasyData` — 📋 DOCUMENTED (deferred)
**File:** `Code.gs`
**Lines:** 1810, 2062, 2142
**Risk:** Theoretical — concurrent executions (web-app `doGet` + a manual run + the trigger) could double-refresh the token or double-send the email

Deferred deliberately: this is a single-user tool on a weekly Monday trigger; Supabase persistence is idempotent (upsert with `resolution=merge-duplicates`); the only real-world effect would be a rare duplicate email. Adding `LockService` to the auth path adds failure modes (lock timeouts) with negligible payoff. Documented for awareness.

### 8. OAuth Callback `doGet` Has No `state` Parameter — 📋 DOCUMENTED (deferred)
**File:** `Code.gs`
**Lines:** 1957–1980 (`getYahooAuthUrl_`), 2007–2060 (`doGet`)
**Risk:** Low — no CSRF `state` round-trip on the OAuth callback

For a single-user personal tool the practical risk is minimal (the attacker would need to trick the sole owner into completing an auth flow). Noted for defense-in-depth; a `state` nonce stored in `CacheService` and validated in `doGet` is the standard fix when this is prioritized.
