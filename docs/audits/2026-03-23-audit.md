## 🟠 High Priority (This Week)

### 1. XSS / HTML Injection in Email Body
**File:** `Code.gs`
**Lines:** 1283, 1503, 1734
**Risk:** User-controlled or API-sourced strings (league name, team name, error messages) are interpolated directly into HTML strings without escaping. If any of these contain `<`, `>`, or `&`, the HTML structure breaks. A malformed team name or API error message containing `</h2><script>` would execute in some email clients.

**Affected locations:**
- `Code.gs:1292` — `leagueName` inserted into `<h2>` tag
- `Code.gs:1503` — `err.message` inserted directly into `<p>` tag
- `Code.gs:1734` — `err.message` inserted into league error section in `pullFantasyData`

**Recommended Fix:**
Add an HTML escaping helper and use it consistently:
```javascript
function escapeHtml_(str) {
  return String(str || '')
    .replace(/&/g, '&amp;')
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;')
    .replace(/"/g, '&quot;');
}
```
Then use `escapeHtml_(leagueName)`, `escapeHtml_(err.message)` etc. when building the HTML body.

**Why High Priority:** While this is a personal tool and the attack surface is very limited, malformed API responses or special characters in league/team names (e.g., `Bosses & Bears`) will break the HTML structure of the email, causing it to render incorrectly. The `&` case is likely to occur in practice.

### 2. Redundant Duplicate API Calls in buildLeagueSnapshot_
**File:** `Code.gs`
**Lines:** 1277–1507
**Risk:** `buildLeagueSnapshot_` calls both `getWeekBenchSummary_(completedWeek, leagueKey)` (line 1433) and implicitly `getTopWaiverPickupsForWeek_` which calls `getWeekStartedPlayerKeys_` (line 1236). Both `getWeekBenchSummary_` and `getWeekStartedPlayerKeys_` independently fetch the full `/league/{key}/teams;week=N;out=roster` endpoint. This means the same large payload is fetched twice per run per league.

Additionally, `pullFantasyData` calls `getLeagueStandings_` again at line 1722 even though `buildLeagueSnapshot_` already called it at line 1281 — a third duplicate call.

**Recommended Action:**
1. Refactor `getWeekBenchSummary_` and `getWeekStartedPlayerKeys_` to accept a pre-fetched roster data argument, or combine them into a single `getWeekRosterData_` function that returns both bench and started info.
2. In `pullFantasyData`, reuse the standings data already fetched inside `buildLeagueSnapshot_` rather than refetching.

**Why High Priority:** Each extra API call increases execution time and risks hitting GAS URL Fetch quotas (20,000/day for consumer accounts). For a 12-team league this is one extra large request per league per run.

**File:** `Code.gs`
**Lines:** 1891
**Risk:** The OAuth callback receives `params.code` and immediately uses it in a token exchange without any length or format validation. A crafted request to the Web App URL with a garbage `code` parameter would trigger a Yahoo API call that returns a 400, which is handled correctly — but it wastes an API call and logs an error that looks like a real failure. This is low risk since the Web App URL is not widely known.

**Recommended:** Add `if (params.code && params.code.length > 10)` as a basic sanity check.
